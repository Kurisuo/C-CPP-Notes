## 4 Properties of a Good lock

 - Must be MUTUALLY EXCLUSIVE: At most one thread is inside the critical region at any given moment
 - Generality: Must be able to support ANY number of threads
 - Independence: A thread outside of the critical region must NOT block another thread from entering the critical region
 - Bounded Waiting: No thread should be stuck/forced into waiting before entering its critical region

## Strict Alternation (Taking Turns)

 -  We use a variable *TURN* which indicates which thread must go next
 - The thread whose turn is not active is stuck in a *SPIN LOCK*

** Refer to alternation.c for example **
 - We can assume the operations are Atomic because we are *setting* rather than incrementing
 - However, say we have a case where a thread is in its noncritical state but is stuck there doing a long operation, this would violate principles 3 and 4, keeping the other thread blocked from accessing its critical region.

## Dekker's Algorithm

- Each thread proclaims interest via a flag 
- the TURN variable does not enforce alternation here but rather is used as a tiebreaker when multiple threads attempt accessing the critical region

- Assumes Atomic single word reads/writes; turn never returns garbage from a corrupt write
- Coherent memory: when thread 0 writes, thread 1 will eventually see that write
- COnsistent memory ordering: the threads see memory operations in the order in which they appear in the program

! Modern CPUs can delay writes and are aggressive on reordering, we require memory barries !

-> LOCK CMPXCHG (x86)| LDREX / STREX (ARM)

## Lamport's Algorithm

- Works with N threads through mutual exclusion.
- Uses the Deli Counter analogy:
    - Thread Walks in, takes a number, and is assisted based on the next lowest number
    - WHen the thread is assisted (enters the critical region), drop the number back down to 0
- Utilizes two arrays
- Each Thread (i) owns two variables. No other varible may write to (i), however, they can READ into them
- Number[i] - ticket number
- Entering[i] - Flag indicating chossing a number[i]
- NUMBER PICKING IS NOT ATOMIC! Two threads can have 
    - The tiebreaker is based on lexicographic order
    - the thread with the smaller ID wins


## Reader - Writer Locks

    - Majority of programs want to perform READ-ONLYs rather than writes, thus they must be EXCLUSIVE
    - Multiple readers hold the lock at the same time
    - Only one writer may hold a lock exclusively -- as in no readers and no other writers other than itself
    - By rwlocking the read heavy workload, we dramatically increase throughput and as such parallelism
    - We choose rwlock over mutexing when we need to perform a high number of reads compare to writes 

## Monitors

    - High Level synchronization primitive
    - Multiple languages do not support Monitors
        - Multiple entry points
        - AT MOST ONE THREAD inside the monitor at all times
        - Enforces mutual exclusivity
        - All variables inside a monitor are protected from simultaneous access
    - The implementation of Monitors is typically handled by the language/compiler
    - A Single thread can be in multiple functions,but no other threads can use said functions at the same time
    - We use Condition Variables within Monitors
        - Wait() suspends the thread until signaled
        - Signal() wakes up the thread waiting on said condition variable
            - if not thread is waiting then signal has no effect
    - We are not required to mark critical region boundaries with mutexes

    Which thread gets the monitor after signal()? -> 

    - Mesa Semantics:
        - Awoken thread placed into a ready queue. competed for the lock and runs at a later convenient time
    - Hoare Semantics: 
        - Awoken thread runs immediately. the signaling thread pauses. ownership of monitor lock and CPU passed to the waiter
